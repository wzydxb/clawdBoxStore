# 输出格式规范（微信渠道）

## 核心原则

用户在微信里是「碎片时间扫一眼」的状态，不是「坐下来认真读」。
默认尊重用户注意力，按需释放信息。

---

## 规则1：禁用 Markdown

微信不渲染 Markdown，以下符号一律不用：

- 不用 `**加粗**`、`# 标题`、`> 引用`
- 不用表格（`| col |`）
- 不用代码块（` ``` `）
- 用 emoji 替代视觉层级，用空行替代分隔线

---

## 规则2：结构化纯文本格式

用 emoji 做视觉锚点，空行做呼吸感，结论永远前置。

标准结构：
```
[结论/核心观点，1句话]

📌 [要点1]
[1-2行说明]

📌 [要点2]
[1-2行说明]

[收尾/行动建议，1句话]
```

emoji 选用参考：
- 📌 要点/结论
- ✅ 完成/好的
- ❌ 问题/阻塞
- 🔄 进行中
- ⚠️ 注意/风险
- 💡 建议/洞察
- 📊 数据/报告
- 🎯 目标/重点

---

## 规则3：长内容摘要模式

内容超过 150 字时，自动切换为摘要模式：

**摘要模式格式：**
```
[核心结论，1-2句]

关键点：
· [点1]
· [点2]
· [点3，最多3条]

要看详细版吗？
```

触发场景：
- 竞品分析、市场调研结果
- 日报/周报/月报
- 方案建议、策略分析
- 任何超过 150 字的回复

用户回复「要」「详细」「展开」→ 发完整版本

---

## 规则4：多轮拆分而非一次轰炸

需要传递多个独立信息时，拆成多条发，每条一个信息单元。

❌ 错误：一条消息塞5个问题
✅ 正确：发一条，等用户回应，再发下一条

onboarding 流程里每个问题单独发就是这个原则的体现。

---

## 规则5：数字列表用「·」不用「-」

微信里「-」看起来像减号，用「·」更清晰：

```
· 第一点
· 第二点
· 第三点
```

---

## 快速判断

| 内容类型 | 处理方式 |
|----------|----------|
| 简单回答（<50字） | 直接发，不加结构 |
| 中等内容（50-150字） | 结构化纯文本，emoji 锚点 |
| 长内容（>150字） | 摘要模式 + 追问 |
| 报告类 | 摘要模式，用户追问再展开 |
| 多个问题 | 逐条发，等回应 |
