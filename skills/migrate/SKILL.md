# 迁移协议 — 从其他龙虾继承记忆

## 触发时机

以下任一情况触发：
- onboarding 第1步，用户提到「之前用过XX龙虾/AI Bot」
- 用户说「我从XXX换过来的」「之前在用XXX」
- 用户主动说「帮我迁移一下」

---

## 第一步：了解用户从哪里来

发一条消息：

```
你之前用的是哪个龙虾？（openclaw / easyclaw / qclaw / autoclaw，或者其他的）

用了多久，大概帮你做过哪些事？
```

根据用户描述的使用时长和任务类型，在内部判断迁移价值：
- 用了 **1个月以上** 且有明确习惯 → 值得深度迁移，进入完整流程
- 用了 **不到1个月** 或描述模糊 → 做轻量提取，重点迁移角色设定

---

## 第二步：引导用户提供记忆原料

根据用户的情况，灵活选择以下一种或几种方式收集原料：

### 方式A：粘贴聊天记录片段（最推荐）

```
把你觉得最能代表「你们之间默契」的几段对话粘贴给我——
比如它做了哪件事你特别满意，或者你纠正过它哪个地方。

不用整理格式，直接粘就行。
```

### 方式B：让用户描述旧 Bot 的人设/记忆

如果用户的旧 Bot 有人设说明或记忆摘要，让他粘贴：

```
如果你之前设置过它的人设，或者它有记忆文件，粘贴过来。
我直接继承，不用重新养。
```

### 方式C：用户口头总结

如果用户没有记录，引导口头描述：

```
没有记录也没关系，说说它给你留下印象最深的几件事：

· 它做得最好的是什么？
· 你最常让它帮你做什么？
· 有没有你特别习惯的回答方式，比如格式、长短、语气？
· 有没有让你不满意、纠正过它的地方？
```

---

## 第三步：分析提取（内部执行，不展示给用户）

拿到原料后，按以下维度提取信号：

### 提取维度

**1. 工作习惯**
- 用户最常做哪类任务？（报告/分析/查数据/写文案/整理文件）
- 任务的典型触发方式是什么？（直接描述 / 关键词 / 习惯性用语）
- 有没有固定的工作节奏？（日报时间、周报习惯等）

**2. 沟通偏好**
- 喜欢长回复还是短回复？
- 喜欢列表还是段落？
- 语气：正式/随意/直接/带温度？
- 结论前置还是先铺垫？
- 有没有被反复纠正的格式问题？（这是强信号）

**3. 专业背景**
- 行业特征（用了哪些专业词汇）
- 业务场景（B端/C端、规模、团队情况）
- 常用工具/数据源

**4. 价值观/决策风格**
- 做决策快还是慢？
- 更信数据还是感觉？
- 对风险的态度？

**5. 旧 Bot 的不足（差异化机会）**
- 用户抱怨过什么？
- 旧 Bot 哪里做得不够好？→ 这是我们超越的机会点

---

## 第四步：写入记忆（静默执行）

提取完成后，静默写入以下位置：

### 写入 USER.md Twin Playbook

```python
import datetime

additions = {
    "沟通规则": [
        # 从提取结果填入，例如：
        # "- 回复长度：简洁优先（迁移自旧Bot习惯，{date}）",
        # "- 格式：默认段落，不用列表",
    ],
    "任务执行规则": [
        # 例如：
        # "- 报告类任务直接出图，不用先问要不要",
        # "- 数据分析默认用pandas，输出核心指标",
    ]
}

content = open('/root/.hermes/USER.md').read()
date_str = datetime.date.today().isoformat()

twin_section = "### 沟通规则"
for rule in additions["沟通规则"]:
    rule_with_date = rule.replace("{date}", date_str)
    if rule_with_date not in content:
        content = content.replace(twin_section, f"{twin_section}\n{rule_with_date}")

open('/root/.hermes/USER.md', 'w').write(content)
print("USER.md updated with migration data")
```

**实际执行时**：用从聊天记录里真实提取的偏好替换注释示例，不要写空占位符。

### 写入 fact_store（偏好记忆持久化）

对每条高置信度偏好，调用：
```
fact_store(action="add", content="迁移偏好：<具体描述>（来源：旧Bot聊天记录）", category="user_pref", tags="preference_signal,migrated,<角色命名空间>")
```

### 写入 TASKLOG.md（如果用户提到了进行中的项目）

如果用户提到旧 Bot 在帮他做某个未完成的项目，写入：
```
| [项目名] | 进行中（迁移自旧Bot） | [日期] | [用户描述的进展] |
```

## 第五步：告知用户并主动续上旧工作

迁移完成后发：

```
好，我读完了。

[用1-2句话总结提取到的最关键信息，让用户感到「它真的懂了」]
比如：你常让它写日报，喜欢短平快的格式，数据分析这块主要用电商的销售数据。

这些我都记住了，不用重新说。

[如果发现旧Bot的明显不足，可以说一句]
比如：它之前做报告总是文字版，我这边可以直接出图，下次试试？

[如果有未完成项目]
你之前在做[项目名]，要继续吗？
```

## 差异化承诺（内部逻辑，不直接说出来，用行动体现）

比旧龙虾更好的具体体现：

| 旧龙虾的常见局限 | 我们的改进 |
|----------------|-----------|
| 记忆重置，换会话就忘 | fact_store 永久记忆，跨会话不丢 |
| 只能文字回复 | 结构化内容直接出图（canvas） |
| 没有主动性，等用户问才答 | 主动出击，检测 TASKLOG 主动推进 |
| 技能单一，没有角色深度 | 行业大佬思维原型 + 专属角色工作流 |
| 无法真实操作电脑 | 浏览器自动化、文件处理、代码执行 |
| 数据获取靠用户粘贴 | 直接抓取：天眼查/热榜/股票/政策 |
| 工作流固定，无法自进化 | Workflow Adapt 自动学习用户习惯 |

**不要直接说「我比它好」，用行动体现。** 用户自己会感受到。

---

## 注意事项

- 用户粘贴的聊天记录可能格式混乱，重点提信息，不要纠结格式
- 如果记录量很大（超过 2000 字），优先提取「用户纠正 Bot」的片段——这是最强的偏好信号
- 不要承诺「完全继承旧 Bot 的所有功能」，只承诺「你的习惯我记住了，不用重新养」
- 如果用户的旧 Bot 有些功能我们暂时没有，不主动提，等用户问到时再说
