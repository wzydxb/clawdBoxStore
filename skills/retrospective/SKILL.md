# 总结反思优化 · 经验沉淀

## 核心逻辑

```
事件/任务 → 结构化复盘 → 提炼经验 → 写入 Twin Playbook → 下次执行更好
```

这不是写报告，是在给用户的"数字大脑"升级。

---

## 触发条件

| 用户说 | 行动 |
|--------|------|
| 复盘/反思/总结 | 启动复盘流程 |
| 这件事做完了/搞定了 | 主动问：要复盘一下吗？ |
| 我学到了什么/经验沉淀 | 直接进入经验提炼 |
| 优化一下/下次怎么做更好 | 进入优化建议阶段 |

---

## 复盘流程（4步）

### Step 1：还原事件

用 2-3 个问题快速还原上下文（不要问太多）：
- 「这件事的背景是什么？目标是什么？」
- 「实际发生了什么？结果怎样？」
- 「和预期相比差了多少？」

如果用户已经描述清楚，跳过问题直接分析。

### Step 2：结构化复盘

用 **WWWH 框架** 分析：

```
🔍 复盘 · {事件名} · {日期}

📌 背景
{一句话说清楚：什么情境、什么目标}

✅ 做得好的
- {1}: {为什么有效？下次可以复用}
- {2}: ...

❌ 做得不好的
- {1}: {根本原因是什么？不是表面原因}
- {2}: ...

🔑 关键转折点
{哪个时刻决定了结果的走向？}

💡 如果重来一次，我会
- {具体的行动改变，不是泛泛而谈}
```

### Step 3：提炼经验规则

把复盘结论转化为**可执行的经验规则**，格式：

```
遇到 [情境] 时，应该 [做法]，因为 [原因]。
避免 [反模式]，曾经因为这样导致 [后果]。
```

规则要满足：
- 够具体（不是"要认真"这种废话）
- 有情境触发条件
- 来自真实经历

### Step 4：写入 Twin Playbook

将提炼的经验规则写入 `~/.hermes/USER.md` 的 `## Twin Playbook` 区块：

```bash
# 读取当前 USER.md，追加经验条目
```

格式：
```markdown
### 经验 · {日期} · {事件标签}
- 规则1：遇到XX，应该YY
- 规则2：避免ZZ，因为...
```

写入后告知用户：
「好的，这条经验已经记下来了。下次遇到类似情况，我会提醒你。」

---

## 主动触发场景

当用户任务状态变为「已完成」时，等1-2轮对话后主动问：

「{任务名} 搞定了，要不要花2分钟复盘一下？有没有什么经验可以沉淀？」

不强迫，用户说不用就算了。

---

## 周期性总结触发

每周五 or 用户说「本周总结」时，额外执行：

**模式识别**：扫描本周的经验条目，看是否有重复出现的模式：
- 同类问题出现2次以上 → 说明这是系统性问题，需要建立SOP
- 同类成功出现2次以上 → 说明这是可复制的方法论，强化记录

输出：
```
🔁 本周模式识别
- 重复出现的挑战：{描述}  → 建议建立标准流程
- 重复有效的方法：{描述}  → 已加入你的方法论库
```
