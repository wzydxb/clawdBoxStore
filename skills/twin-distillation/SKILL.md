# Twin Distillation — 偏好蒸馏协议

你的任务是从当前会话的对话内容中提炼用户的行为偏好，并更新 USER.md。

## ⚠️ 核心约束：原型骨架不可覆盖

SOUL.md 中 `<!-- ARCHETYPE_LOCK -->` 和 `<!-- /ARCHETYPE_LOCK -->` 之间的内容是职业原型骨架，**twin-distillation 绝对禁止修改**。

蒸馏的目标是在骨架之上叠加用户血肉，不是替换骨架：

| 层级 | 归属 | 写入位置 | 能否蒸馏修改 |
|------|------|----------|------------|
| 骨架 | 职业原型（onboarding 写入） | SOUL.md `思维框架` 区域 | ❌ 禁止 |
| 血肉 | 用户个人特质（蒸馏积累） | USER.md Twin Playbook | ✅ 只写这里 |

**用 `skill_manage` 时**：只允许写角色命名空间下的工作流技能文件（如 `pm/workflow`），禁止操作 SOUL.md 本身。

---

## 触发条件

在以下任一情况发生时执行本技能：

1. **明确偏好表达**：用户说「就这样」「喜欢这种」「下次也这样做」「记住这个」
2. **行为纠正**：用户说「不用这么正式」「直接说结论就好」「不要用列表」
3. **隐性信号**：用户对某种回答方式重复表示认可（连续3次以上）
4. **会话轮次**：当前会话超过5轮对话，主动分析是否有新模式

---

## 蒸馏步骤

### Step 1：回顾会话

读取当前会话的对话内容，重点关注：
- 用户对你回答的反应（接受/修改/拒绝）
- 用户自己的表达风格（用词、句子长度、正式程度）
- 用户做决策的方式（快速拍板/反复权衡/要数据支撑）
- 用户处理任务的优先级逻辑

### Step 2：识别信号

**沟通风格信号：**
- 用户喜欢简短还是详细的回答？
- 喜欢列表还是段落？正式还是随意？
- 喜欢先结论还是先铺垫？

**工作方式信号：**
- 任务优先级排序方式？
- 需要数据支撑还是凭感觉决策？
- 喜欢一次搞定还是分步迭代？

**价值观信号：**
- 什么让用户特别认可？什么让用户不满意？

### Step 3：判断置信度

- **高置信度（直接写入）**：用户明确表达，或连续3次以上一致反应
- **低置信度（写入 Preference Signals）**：只观察到1-2次，尚不确定

### Step 4：更新 USER.md（唯一允许写入的位置）

读取 `~/.hermes/USER.md`，按以下规则**增量追加或修改**，不覆盖已有内容：

```
高置信度偏好 → 写入「## Twin Playbook」章节
低置信度偏好 → 在「## Twin Playbook」末尾追加「### 待验证信号」子章节
兴趣/专长发现 → 写入「## Joey 特质（血肉）」→「### 核心能力」
工作方式更新 → 写入「## Joey 特质（血肉）」→「### 工作方式」
```

定位方式：用 `grep -n "^## "` 找到章节行号，在对应章节末尾追加，不要覆盖整个章节。
```markdown
### 沟通执行规则
- 回答长度：优先简洁，用户明确要详细时才展开（确认于 YYYY-MM-DD）
- 格式：默认不用列表，除非对比多个选项
- 结论前置：先给结论，再解释原因
```

**冲突处理**：新观察与已有规则矛盾时，标注「待验证」而不是直接覆盖。

### Step 5：稳定偏好固化（可选，需确认）

每次蒸馏到偏好信号时，先用 fact_store 记录：
```
fact_store(action="add", content="偏好信号：<描述>", category="user_pref", tags="preference_signal,<角色命名空间>")
```

然后查询同类已有记录数：
```
fact_store(action="search", query="preference_signal", category="user_pref")
```
读取返回 JSON 中的 `count` 字段（而非自己数列表）。

若 `count >= 2`，告知用户并询问是否固化到技能文件。确认后执行：
```
skill_manage(action="edit", skill="<角色命名空间>/workflow", append="## 用户偏好规则\n- <偏好描述>（固化于 YYYY-MM-DD）")
```
**绝不修改 SOUL.md，只写角色命名空间下的工作流技能文件。**

### Step 6：更新 Last Updated 字段

将 USER.md 头部的 `Last Updated` 改为今天日期。

## Workflow Adapt — 工作流自进化

> **目标**：不依赖用户主动说偏好，通过观察"实际执行路径 vs workflow 预设路径"的偏差，自动把有效偏差固化回 workflow。

### 触发条件

满足以下任一条件时，在偏好蒸馏结束后**追加执行** Workflow Adapt：

1. 本次会话完成了一个完整任务（有明确的开始 → 执行 → 结果）
2. 执行过程中跳过了 workflow 中的某个步骤，且用户没有抱怨
3. 执行过程中增加了 workflow 中没有的步骤，且用户表示满意
4. 用户说「下次直接这样做」「不用走那个流程」「这个步骤没必要」

### Adapt 步骤

#### Step A：重建执行路径

回顾本次会话，列出实际执行的步骤序列：
```
实际路径：[步骤1] → [步骤2] → [步骤3] → ...
```

读取当前角色的 workflow skill（`skill_view("<role_dir>/workflow")`），提取 workflow 预设的标准路径。

#### Step B：识别有效偏差

对比两条路径，找出偏差点，按类型分类：

| 偏差类型 | 描述 | 是否值得固化 |
|---------|------|------------|
| 跳过步骤 | workflow 有但实际没做，结果仍好 | ✅ 候选 |
| 新增步骤 | workflow 没有但实际做了，用户满意 | ✅ 候选 |
| 顺序调整 | 步骤顺序与 workflow 不同，效果更好 | ✅ 候选 |
| 工具替换 | 用了不同的工具/数据源，效果更好 | ✅ 候选 |
| 失败回退 | 走了弯路后回到 workflow，无价值 | ❌ 忽略 |

**只记录用户没有抱怨、且任务成功完成的偏差。**

#### Step C：记录偏差到 fact_store

每个有效偏差记录一条：
```
fact_store(
  action="add",
  content="workflow偏差：<角色>/<任务类型> — <偏差描述>（如：跳过了竞品分析直接写PRD，用户满意）",
  category="workflow_adapt",
  tags="workflow_delta,<角色命名空间>,<任务类型>"
)
```

#### Step D：查询积累次数

```
fact_store(action="search", query="<偏差描述关键词>", category="workflow_adapt")
```

读取返回 JSON 的 `count` 字段：
- `count == 1`：首次观察，静默记录，不打扰用户
- `count >= 2`：同一偏差出现2次以上，进入固化流程

#### Step E：提议固化（count >= 2 时）

告知用户，语气轻松，一句话：
> 「我注意到你做[任务类型]时通常会[偏差描述]，要把这个加到我的工作流里吗？」

用户确认后，执行 patch：
```
skill_manage(
  action="edit",
  skill="<角色命名空间>/workflow",
  append="## 用户定制路径\n- [任务类型]：<patch描述>（固化于 YYYY-MM-DD）"
)
```

用户拒绝时，记录一条 `workflow_adapt_rejected` 标签，下次同类偏差不再提议。

#### Step F：注册新 skill（如果新增步骤是调用了新工具）

如果偏差是"新增了一个 workflow 里没有的 skill 调用"，且固化成功，同步在 AGENTS.md Skill Registry 中注册：
```
skill_manage(
  action="edit",
  skill="agents",  # 即 AGENTS.md
  append="| <skill名> | `skill_view(\"<path>\")` | <一句话描述> | <触发词> |"
)
```

- 保持 USER.md 简洁，一条规则一行
- Workflow Adapt 是静默后台操作，count < 2 时不打扰用户
