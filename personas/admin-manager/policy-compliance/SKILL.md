# 制度文件管理

## 适用场景
- 新制度起草和发布
- 现有制度定期修订
- 合规检查和审计准备
- 制度宣贯和培训

---

## 制度全生命周期（7 阶段）

```
起草 → 审核 → 审批 → 发布宣贯 → 执行监控 → 定期修订 → 退役归档
```

| 阶段 | 关键动作 | 输出 |
|------|----------|------|
| 起草 | 需求分析、法规调研、责任人指定 | 制度草案 |
| 审核 | 相关方会签、法务审查、风控审查 | 修改意见 |
| 审批 | 分管领导/总经理签批 | 审批记录 |
| 发布宣贯 | 统一发布、全员培训、签收确认 | 培训记录 + 签收单 |
| 执行监控 | 执行检查、例外跟踪、KPI 监控 | 执行报告 |
| 定期修订 | 年度/触发式修订、版本迭代 | 新版本 |
| 退役归档 | 正式废止、替代检查、归档 | 归档记录 |

## ISO 9001:2015 文件控制要求（条款 7.5）

> ISO 9001 用「documented information」统一了「文件」和「记录」。以下是核心控制要求：

| 控制要求 | 说明 |
|----------|------|
| 创建与审批 | 文件发布前必须经过审批 |
| 修订与再审批 | 更新后必须重新审批 |
| 变更标识 | 清楚标识修订了什么、当前是第几版 |
| 使用点可获取 | 需要用到的人能随时拿到最新版 |
| 可读性与可识别 | 文件编号、标题、版本号清晰 |
| 外来文件控制 | 外部法规、客户文件也纳入管控 |
| 防止误用作废版 | 作废文件必须标记或移除 |

**PDCA 循环集成：**
- Plan：制度规划和起草
- Do：发布和执行
- Check：检查执行情况
- Act：修订改进

## 制度起草规范

**制度文件标准结构：**
1. 目的——为什么要有这个制度
2. 适用范围——谁受这个制度约束
3. 术语定义——专业术语解释
4. 职责分工——谁负责什么
5. 具体规定——条款要清晰、可执行、可量化
6. 违规处理——违反后果
7. 附则——生效日期、修订周期、解释权归属
8. 附件——表单模板、流程图

## 合规检查清单

### 年度合规检查

| 检查项 | 检查内容 | 合规标准 |
|--------|----------|----------|
| 制度覆盖率 | 核心业务场景是否都有制度覆盖 | > 90% |
| 版本时效性 | 超过修订周期未更新的制度占比 | 过期率 < 10% |
| 执行符合率 | 抽查实际执行是否符合制度规定 | > 85% |
| 员工知晓率 | 抽查员工对关键制度的了解程度 | > 80% |
| 法规一致性 | 制度是否与最新法律法规一致 | 100% |
| 签收覆盖率 | 新制度发布后员工签收比例 | > 95% |

### 修订触发规则
- 修订周期到期前 30 天 → 提醒责任人启动修订
- 法律法规变更后 15 天 → 检查相关制度是否需要同步更新
- 重大事件后（安全事故、合规处罚）→ 立即检查相关制度
- 组织架构调整后 → 检查职责分工条款

## 典型陷阱

- **制度写了没人看**：发布后必须宣贯，关键制度要考试或签收确认
- **制度过时不修订**：业务变了制度没跟上，执行必然走样
- **制度之间互相矛盾**：多个制度涉及同一事项时要交叉检查
- **没有单一存储库**：制度散落在 SharePoint、邮件、个人电脑中，找不到最新版
- **只有制度没有审批记录**：审计时无法证明制度经过正式审批
